# 智能垃圾分类助手小程序的设计与实现

## 中文摘要

随着生活垃圾分类制度的持续推进，居民在实际投放过程中仍普遍面临分类规则难记忆、查询路径不便捷、分类依据不清晰等问题。传统宣传手册和静态问答虽然能够完成基础科普，却难以满足日常场景下对即时性、便捷性与可解释性的综合需求。针对上述问题，本文面向居民日常垃圾投放场景，设计并实现了一款智能垃圾分类助手小程序。

该系统采用前后端分离架构，前端基于微信小程序原生框架实现轻量化交互，后端基于 Flask 构建业务服务，并结合 MySQL、Redis 与 ECharts 完成数据存储、缓存优化和知识图谱展示。在功能层面，系统集成图像识别、语音辅助查询、文字搜索、知识图谱展示、科普文章浏览和历史记录管理等模块。其中，图像识别部分基于 MobileNetV3 构建垃圾四分类模型；语音与文字模块则共同承担多场景输入支持；结果页进一步结合分类说明、低置信度提示和知识图谱，增强了系统输出的可解释性。

根据项目开发记录与阶段性验收结果，系统图像分类模型在验证集上的准确率达到 85.02%，训练集与验证集规模分别为 46944 张和 5007 张，后端自动化测试共 42 项并已通过，主要业务链路已完成集成验证。研究表明，该系统能够较好满足居民在垃圾分类查询与知识学习方面的实际需求，并可为面向公共服务场景的垃圾分类智能化应用设计与实现提供参考。

**关键词：** 垃圾分类；微信小程序；图像识别；知识图谱；Flask

## Abstract

With the continuous implementation of household waste-sorting policies, residents still encounter practical difficulties during daily disposal, including hard-to-remember classification rules, inconvenient query methods, and insufficient understanding of the rationale behind classification results. Conventional publicity materials and static lookup methods can provide basic guidance, yet they are often inadequate for real-time and explainable assistance in everyday scenarios. To address these issues, this thesis designs and implements an intelligent garbage-classification assistant mini-program for daily residential use.

The system adopts a front-end/back-end separated architecture. The client side is developed with the native WeChat Mini Program framework, while the server side is built with Flask. MySQL, Redis, and ECharts are employed for data storage, cache optimization, and knowledge-graph visualization, respectively. Functionally, the system integrates image recognition, voice-assisted query, text search, knowledge-graph display, science-popularization articles, and query-history management. The image-recognition module is built on a MobileNetV3-based four-category classification model, while text and voice modules jointly provide flexible input support across different usage conditions. In addition, a unified result page, low-confidence prompts, and explanatory knowledge links are introduced to improve usability and interpretability.

According to project records and staged acceptance results, the image-classification model achieved an accuracy of 85.02% on the validation set, with 46,944 training samples and 5,007 validation samples. Meanwhile, 42 backend automated tests passed, and the major business chains completed integration verification. These results indicate that the system can effectively support residents in garbage-classification query and knowledge learning, and it provides a practical reference for the design and implementation of intelligent waste-sorting applications in public-service scenarios.

**Key Words:** garbage classification; WeChat Mini Program; image recognition; knowledge graph; Flask

## 1 绪论

### 1.1 研究背景

随着城镇化进程不断加快，城市生活垃圾产生量持续增长，垃圾分类已成为城市精细化治理和生态文明建设中的重要议题。近年来，我国多个城市相继推进生活垃圾分类制度建设，居民分类意识虽有所增强，但在具体投放环节中，“知道要分类，却难以快速判定如何分类”的问题依然普遍存在。尤其是在家庭、校园和社区等日常场景下，垃圾物品种类繁杂、名称多样，且地方化规则存在差异，这使普通居民在面对具体物品时往往难以及时、准确地完成分类判断。

传统的垃圾分类指导方式主要依赖宣传手册、海报展板、社区宣讲和固定问答清单。这类方式在政策普及阶段发挥了积极作用，但仍存在三方面局限：其一，信息呈现较为静态，难以满足即时场景中的快速查询需求；其二，用户往往只能记住少量典型案例，面对边缘样本或复合物品时仍容易产生混淆；其三，传统方式更多停留在“给出答案”，对分类依据、投放原因和相关知识的解释相对不足，不利于用户形成稳定而可迁移的分类认知。

与此同时，移动互联网与人工智能技术的发展为垃圾分类服务提供了新的实现路径。微信小程序具备轻量化、免安装、易传播和易触达等特点，适合构建面向大众的民生服务应用；图像识别技术可以通过拍照辅助用户快速完成初步判断；语音交互与文字搜索能够进一步降低使用门槛；知识图谱可视化则有助于将分类结果与分类依据、相近物品和投放提示建立联系，帮助用户理解“为什么这样分”。也正因为如此，如何将“识别能力”“查询能力”和“解释能力”整合到同一系统中，成为本课题关注的核心问题。

基于上述背景，本文设计并实现了一套集图像识别、语音辅助查询、文字搜索与知识图谱展示于一体的智能垃圾分类助手小程序，以期为居民提供更便捷、更直观、也更具解释性的垃圾分类服务。

### 1.2 研究意义

本课题的研究意义可从实践价值与方法价值两个层面理解。

从实践层面看，垃圾分类服务的关键难点并不只是“信息有没有被提供”，而在于居民能否在具体投放瞬间快速获得可执行的判断依据。基于此，本文以微信小程序为统一入口，将图像识别、文字检索与语音辅助查询整合到同一交互框架中，能够在家庭、校园、社区等高频生活场景下提供更加便捷的查询路径。同时，系统在返回分类结果的基础上进一步补充分类说明、知识图谱和科普文章，使用户不再局限于机械记忆个别案例，而是逐步建立对分类逻辑的整体理解，这对于提升垃圾分类服务的实用性与持续使用价值具有积极意义。

从方法层面看，本文并未将垃圾分类问题简单处理为单一的图像识别或关键词检索任务，而是尝试构建“多模态输入 + 统一结果组织 + 知识解释扩展”的协同实现路径。具体而言，微信小程序、Flask 后端、MobileNetV3 图像分类模型、Redis 缓存机制与知识图谱展示被整合到同一系统框架之中，使智能识别能力与结果解释能力能够相互配合。对于公共服务场景下的轻量化智能应用设计而言，这种从“给出答案”进一步走向“解释答案”的实现思路，具有一定的参考价值。

### 1.3 国内外研究现状

从现有研究看，垃圾分类相关工作大体可以归纳为三个方向。其一是围绕城市治理与居民行为展开的政策和管理研究，此类研究重点关注分类制度推广、居民参与意愿及社区治理机制，为垃圾分类服务需求的提出提供了现实背景；其二是围绕垃圾图像识别和自动分拣展开的智能识别研究，随着卷积神经网络和迁移学习方法的发展，研究重点逐步由离线识别精度拓展至轻量化模型与部署效率；其三是围绕移动端服务、知识组织与智能问答展开的应用研究，主要关注如何通过信息系统降低公众获取分类知识的门槛。

国外研究起步较早，在垃圾回收体系优化、智能回收设备和视觉识别等方面已经形成较成熟的技术路线。尤其在图像识别方向，研究者普遍关注模型精度、推理效率以及嵌入式部署适配问题，这为移动端垃圾分类辅助系统提供了方法基础。国内研究则更多伴随政策推进而快速增长，既包括分类政策执行与居民行为研究，也包括面向实际使用场景的识别系统、小程序应用和知识服务平台建设。

然而，已有研究和现有产品仍存在两方面不足。首先，不少系统更强调“识别是否准确”或“能否查到结果”，而对分类依据解释、相近物品辨析和知识延展支持不足，用户获得答案后仍可能缺乏迁移应用能力。其次，多数方案在交互上仍偏向单一入口，尚未充分兼顾拍照、文字和语音等不同使用场景的输入需求。由此可见，围绕“快速识别、便捷查询与结果解释”构建统一系统，仍然具有明确的研究空间和应用价值。

### 1.4 研究内容与目标

围绕上述问题，本文以智能垃圾分类服务场景为对象，完成了一套基于微信小程序的垃圾分类助手系统的设计与实现。整体研究内容可概括为以下三个方面：第一，结合居民在日常投放中的实际使用情境，开展系统需求分析与总体方案设计，明确图像识别、语音辅助查询、文字搜索、知识图谱展示、历史记录和科普内容等功能边界；第二，完成小程序前端页面、Flask 后端服务以及基于 MobileNetV3 的垃圾图像分类模块实现，形成从用户输入到结果展示、再到知识拓展的完整业务闭环；第三，围绕系统的工程可用性，进一步设计统一响应格式、JWT 鉴权、Redis 缓存与低置信度提示等机制，并结合模型指标、功能测试、接口测试和性能分析对实现效果进行验证。

基于上述研究内容，本文希望达到以下目标：在交互层面，为居民提供低门槛、可快速上手的垃圾分类查询工具；在技术层面，探索多模态输入与结果解释协同实现的系统方案；在工程层面，构建具备一定可维护性、可扩展性和验证基础的毕业设计原型系统。

### 1.5 论文结构安排

本文共分为七章。第一章阐述研究背景、研究意义、国内外研究现状以及研究内容与目标；第二章介绍系统实现所涉及的相关技术与理论基础；第三章分析系统应用场景及功能、非功能需求；第四章给出系统总体设计，包括架构、模块、业务流程和数据库设计；第五章详细说明前端、后端与图像识别模块的设计与实现；第六章对系统测试过程与结果进行分析；第七章对全文工作进行总结，并提出后续优化方向。

## 2 相关技术与理论基础

### 2.1 微信小程序开发技术

微信小程序是一种依托微信生态运行的轻量级应用形态，具有无需下载安装、入口稳定、传播成本低和用户触达方便等特点。对于垃圾分类这类高频、碎片化、场景依赖较强的民生服务应用而言，用户往往更关注“是否能够立即使用”，而不是是否具备复杂的本地功能配置。因此，相比传统独立 App，小程序更适合作为垃圾分类指导服务的主要载体。

从技术实现看，微信小程序采用逻辑层、视图层与系统能力接口分离的开发模式，其中 WXML 负责页面结构描述，WXSS 负责样式控制，JavaScript 负责交互逻辑与数据请求。平台本身已经提供拍照、图片选择、录音、网络通信、本地缓存和页面跳转等能力，这意味着开发者可以在较低实现成本下完成多页面联动和多输入方式接入。对本文系统而言，小程序不仅满足了移动端轻量部署的需求，也为拍照识别、语音辅助查询、结果展示和历史记录等功能提供了统一而稳定的运行环境。

### 2.2 Flask 后端框架与 RESTful 接口设计

Flask 是 Python 生态中常用的轻量级 Web 框架，具有结构灵活、模块拆分清晰和扩展成本较低等特点，尤其适合需要同时组织 Web 接口、数据库访问和人工智能推理服务的应用系统。对于本文课题而言，后端既要负责用户登录、数据查询和结果封装，又要联动图像模型推理与语音识别服务，因此更需要一种便于快速组织业务逻辑、且能够与 Python 深度学习生态自然衔接的框架。基于这一考虑，Flask 成为较为合适的技术选择。

在接口组织层面，本文采用 RESTful 设计思路，将用户登录、图像识别、语音查询、文字搜索、知识图谱、文章和历史记录等功能拆分为相对独立的 API，并统一使用 JSON 数据结构和业务码返回结果。这样的设计不仅有助于前端保持一致化的数据处理方式，也使后端能够在测试、异常处理和后续扩展时维持较好的模块边界，从而提升整体系统的可维护性。

### 2.3 MySQL 与 Redis 数据管理技术

MySQL 是成熟的关系型数据库，适合管理结构稳定、关系明确且需要长期保存的业务数据。在本文系统中，用户信息、垃圾物品信息、知识关系、查询历史和科普文章等内容都具有较强的结构化特征，因此由 MySQL 负责持久化存储更利于保证数据组织的规范性和后续查询扩展能力。

Redis 属于高性能内存数据库，特别适合处理高频访问和短时重复查询场景。垃圾分类服务在实际使用中存在明显的热点特征，例如常见物品名称会被反复检索，相同图片在联调或重复查询时也可能被多次上传。因此，本文将 Redis 引入为缓存层，对搜索结果和图像识别结果进行暂存，以减少数据库访问压力和模型重复推理开销。MySQL 与 Redis 的组合使系统形成“持久化存储 + 热点缓存加速”的分层数据方案，在数据可靠性与响应效率之间取得较好平衡。

### 2.4 基于 MobileNetV3 的图像分类方法

图像分类是系统智能能力的核心组成部分，其目标是在用户上传垃圾图片后快速给出类别判断。与通用场景下的大规模识别任务相比，垃圾分类辅助系统更强调推理效率、部署成本和移动端适配能力，因此模型选择不能只关注理论精度，还需要兼顾实际工程环境中的资源约束。

MobileNetV3 是面向移动端和轻量化部署场景优化的卷积神经网络结构，能够在参数规模、推理速度和识别性能之间取得较好平衡。基于这一特点，本文选用 MobileNetV3 作为图像分类模型的基础框架，并将识别任务定义为可回收物、有害垃圾、厨余垃圾和其他垃圾四分类问题。结合迁移学习策略，可以在已有视觉特征基础上较快完成针对垃圾图像数据集的模型训练，从而满足系统在识别效果与部署效率之间的双重要求。

### 2.5 语音识别技术原理与应用

语音识别技术的核心任务是将语音信号转换为可处理的文本信息，从而降低用户输入成本并提升人机交互自然度。在垃圾分类场景中，用户常处于“手中拿着待投放物品、无法方便打字”的状态，此时语音输入能够提供比手动检索更顺畅的交互方式。

基于这一特点，本文在系统中引入语音辅助查询能力，通过后端语音识别接口将音频内容转换为文本后，再复用文字搜索链路完成后续分类查询。这样设计的好处在于，一方面扩展了系统输入方式，另一方面避免了重新设计独立结果链路所带来的额外复杂度，使语音模块能够以较低耦合度融入既有系统架构。

### 2.6 知识图谱与 ECharts 可视化技术

知识图谱是一种以“实体 - 关系”结构组织知识的表达方式，能够较直观地呈现对象之间的关联网络。对于垃圾分类服务而言，仅给出类别标签往往不足以帮助用户真正掌握分类逻辑，而将物品、类别、相近物品和投放提示之间的联系组织为图谱，则有助于把零散的答案转化为可理解的知识结构。

本文利用知识图谱组织垃圾物品之间的关联关系，并借助 ECharts 及其微信小程序适配组件完成图谱展示与节点交互。这样做的意义在于，系统输出不再停留于“告诉用户结果”，而是进一步支持“解释结果从何而来、与哪些物品相近、还可以延伸了解什么”，从而增强结果页的可解释性与知识延展能力。

## 3 系统需求分析

### 3.1 系统应用场景分析

本系统面向普通居民在家庭、校园、社区等日常生活场景中的垃圾分类需求。与工业分拣场景不同，普通居民在投放垃圾时通常缺乏系统性的分类知识，也很少愿意花费较长时间逐条查阅规则。因此，用户对系统最直接的期待并不是“获得尽可能复杂的知识”，而是在最短时间内获得明确、可执行的分类建议，并在需要时进一步理解其分类依据和投放注意事项。

从具体使用方式看，系统主要对应三类典型场景。第一类是即时判断场景，即用户面对具体物品时需要立即完成分类，此时更适合通过拍照识别或快捷搜索获得结果；第二类是替代输入场景，即用户手上拿着物品、不便键入文字，希望借助语音辅助完成查询；第三类是延伸学习场景，即用户在得到分类结果后，还希望了解相关物品关系、分类原因以及投放提示。基于这些场景，系统设计必须同时兼顾“查询效率”“交互便利”和“结果可解释”三个方面。

### 3.2 功能需求分析

结合系统面向的实际场景，可以将功能需求概括为“便捷输入、准确返回、持续解释”三个层面。具体而言，系统需要满足以下功能需求：

1. 图像识别需求。系统应支持用户拍照或从相册选图上传，并返回垃圾分类结果、识别置信度、分类说明与投放建议，以满足“看到物品即可查询”的即时需求。
2. 语音辅助查询需求。系统应支持将语音内容转换为文本后继续查询，并在识别失败时给出友好提示与回退方案，降低特殊场景下的输入难度。
3. 文字搜索需求。系统应支持关键词、别名和模糊搜索，返回候选物品列表，并提供详情或结果页入口，以便用户在无法拍照时仍能完成快速检索。
4. 结果解释需求。系统不应只返回分类结果，还应提供分类依据、投放建议、候选结果和低置信度提示，帮助用户理解结果而非机械记忆答案。
5. 知识拓展需求。系统应支持知识图谱展示和科普文章浏览，使用户能够在完成一次查询后继续了解相关物品关系和垃圾分类知识。
6. 历史记录需求。对于已登录用户，系统应能够保存并展示图像、语音和文字三种查询路径下的历史记录，便于后续查看与行为留痕。

### 3.3 非功能需求分析

除功能实现外，系统还需满足如下非功能需求，以保证整体可用性、稳定性与可维护性：

1. 易用性。页面结构应简洁清晰，操作路径应尽量短，提示信息应准确直观，避免用户在查询过程中产生额外理解成本。
2. 实时性。文字搜索与图像识别应在可接受时间内返回结果，确保系统能够适应日常投放环节中“边查边分”的使用节奏。
3. 安全性。受保护接口需通过 JWT 鉴权，敏感配置与密钥信息不得在前端暴露，用户历史记录等数据需在授权前提下访问。
4. 稳定性。系统在 Redis、语音识别等外部依赖不可用时，应具备合理的降级处理能力，尽可能保证核心查询链路不被阻断。
5. 可维护性。系统应采用清晰的模块划分和统一的数据结构，便于后续开展接口维护、功能迭代与测试扩展。
6. 可扩展性。系统架构应支持后续扩充垃圾物品数据、知识关系、识别模型能力与部署形态，为持续优化预留空间。

## 4 系统总体设计

### 4.1 系统设计目标

本系统的总体设计目标可以概括为“便捷查询、智能识别、结果可解释、工程可扩展”。具体而言，系统既要面向居民日常投放场景提供快速、低门槛的查询服务，也要在后台保持较清晰的模块边界和较稳定的服务链路，从而兼顾用户体验与工程实现的完整性。围绕这一目标，系统设计重点落在三个方面：一是通过多种输入方式降低使用门槛；二是通过统一结果页与知识图谱提升结果可解释性；三是通过前后端分层、缓存机制和统一接口格式提高系统的可维护性与扩展性。

### 4.2 系统总体架构设计

系统采用前后端分离架构，整体可划分为前端展示层、后端业务层、数据存储层和智能服务层四个部分。前端展示层基于微信小程序完成用户输入采集、页面跳转和结果展示；后端业务层基于 Flask 负责接口组织、请求分发、权限控制和业务编排；数据存储层由 MySQL 与 Redis 组成，分别承担持久化存储与热点缓存任务；智能服务层则包括图像识别模型与语音识别服务，为系统提供多模态查询能力。

从数据流角度看，用户在前端发起图像、语音或文字查询请求后，请求首先进入 Flask 后端；后端再依据请求类型调用搜索服务、图像识别服务或语音识别服务，并结合数据库与缓存进行结果拼装；最终，系统以统一响应格式将结果返回前端，再由结果页、知识图谱页或文章页继续承接后续交互。这样的层次划分既有利于控制系统复杂度，也便于后续针对单一层次开展性能优化和功能扩展。

### 4.3 系统功能模块划分

系统主要由用户认证模块、图像识别模块、语音查询模块、文字搜索模块、结果展示模块、知识图谱模块、科普文章模块和历史记录模块组成。其中，用户认证模块负责登录状态建立与受保护资源访问控制；图像识别、语音查询和文字搜索模块分别对应三种不同的输入入口；结果展示模块负责统一承接分类结果与详情信息；知识图谱与科普文章模块承担结果解释与知识延展功能；历史记录模块则用于保留用户的查询轨迹。

这些模块在功能上彼此分工明确，但在数据结构和交互流程上保持统一。三种查询入口最终都会汇聚到同一结果展示逻辑，再进一步延伸到知识图谱、文章与历史记录页面，从而形成较为稳定的业务闭环。

### 4.4 系统业务流程设计

系统主要业务流程如下：

1. 用户登录流程。前端调用微信登录能力获取临时凭证，后端根据 `code` 换取 `openid` 并签发 JWT，为历史记录等受保护接口提供访问依据。
2. 图像识别流程。用户拍照或选图后发起上传，后端完成文件校验、图片存储、缓存判断、模型推理和结果拼装，再由前端跳转至结果页进行展示。
3. 语音查询流程。系统将语音内容转换为文本后复用文字搜索服务；若识别失败，则向用户提供切换到文字查询的回退路径。
4. 文字搜索流程。前端提交关键词，后端依次通过缓存、全文检索和模糊匹配返回候选物品列表，用户再继续进入详情页或结果页。
5. 知识拓展流程。用户在得到分类结果后，可进一步进入知识图谱页与文章页，完成由“获得答案”到“理解原因”的延伸学习。

需要指出的是，以上流程并非彼此孤立，而是共同构成以结果页为中心的业务闭环。无论用户通过图像、语音还是文字入口发起查询，最终都会汇聚到统一的结果展示与知识拓展页面。这样的流程设计既提高了交互一致性，也使后端能够围绕统一结果结构组织接口输出，从而降低多入口系统在实现和维护上的复杂度。

### 4.5 数据库设计

系统核心数据表主要包括 `users`、`categories`、`garbage_items`、`knowledge_relations`、`query_histories` 和 `articles`。其中，`categories` 与 `garbage_items` 构成一对多关系，用于表达垃圾类别与具体物品之间的归属关系；`users` 与 `query_histories` 构成一对多关系，用于记录用户的查询行为；`garbage_items` 与 `knowledge_relations` 则形成以物品为中心的自关联结构，用于支撑知识图谱展示与相关物品联动。

从数据职责看，`categories` 主要保存类别名称、颜色和投放说明等类别级信息，`garbage_items` 主要保存垃圾物品名称、别名、分类依据和投放提示等物品级信息，`knowledge_relations` 用于描述物品与物品之间的关联关系，`query_histories` 用于记录不同查询方式下的用户行为轨迹，`articles` 用于承载科普内容。这样的数据库设计既能够满足系统当前的基础查询需求，也为后续开展知识扩展、行为分析和个性化服务提供了数据基础。

### 4.6 接口设计

系统统一采用 `/api/v1` 作为接口前缀，并使用 `{code, message, data}` 作为响应结构。主要接口包括用户登录接口、健康检查接口、图像识别接口、语音查询接口、文字搜索接口、垃圾详情接口、知识图谱接口、文章接口和历史记录接口。统一的接口风格不仅便于前端进行一致化处理，也有利于在测试阶段对不同接口使用统一的断言逻辑，从而降低接口维护成本。

## 5 系统详细设计与实现

### 5.1 小程序前端设计与实现

系统前端基于微信小程序原生框架开发，主要负责用户输入采集、页面跳转、结果展示以及与后端接口的交互。整体页面结构围绕“首页统一入口、查询方式多样、结果集中展示、知识内容可追踪”的思路展开，尽量缩短用户从发起查询到获得结果的操作路径，并使不同查询方式在视觉与交互层面保持一致。

#### 5.1.1 首页模块实现

首页设置拍照识别、语音查询、文字搜索和历史记录四个主要入口，并增加“今日科普”和分类指南区域。页面加载时会执行健康检查、用户欢迎语同步和推荐文章加载等操作，使首页不仅承担功能分发作用，也承担内容引导和系统状态提示作用。对于初次使用者而言，这种布局有助于其迅速理解系统的主要能力与使用路径。

#### 5.1.2 拍照识别模块实现

拍照识别页支持拍照和相册选图两种输入方式。为了减少传输体积与等待时间，前端在上传前会对图片进行缩放和压缩处理，再通过文件上传接口将图片提交至后端。识别成功后，系统将结果临时写入本地存储并跳转至结果页展示，以避免将过长结果参数直接拼接在页面跳转路径中，从而提升页面间传参的稳定性。

#### 5.1.3 语音查询模块实现

语音查询页当前主要采用语音辅助输入的交互方式。用户先通过语音转文字获得查询文本，系统再调用搜索接口返回候选垃圾物品列表；若识别或查询失败，则给出友好提示，并引导用户切换到文字方式继续完成查询。这样的设计使语音能力更多承担“降低输入门槛”的角色，而不是单独构成一条与文字搜索完全割裂的业务链路，因此更符合当前系统的实现状态。

#### 5.1.4 文字搜索模块实现

文字搜索页支持关键词输入、防抖搜索和结果列表展示。页面在用户停止输入后自动发起查询请求，以减少频繁请求带来的资源消耗；同时，系统在搜索结果中提供继续进入详情页或结果页的入口，方便用户在模糊检索后进一步查看分类说明与知识拓展内容。与图像和语音入口相比，文字搜索在系统中承担更稳定、更通用的基础查询角色。

#### 5.1.5 结果展示与知识图谱页面实现

结果页统一承接图像识别结果与垃圾详情查询结果，集中展示分类名称、置信度、投放建议、相关物品以及低置信度提示。通过将三种查询入口的输出统一收敛到同一结果页，系统能够显著降低用户在不同入口之间切换时的认知负担。知识图谱页则根据物品编号请求后端图谱数据，并支持节点点击联动切换，使用户能够由单一分类结果进一步过渡到关联知识的理解。

#### 5.1.6 历史记录与文章页面实现

历史记录页用于展示用户通过图像、语音和文字三类方式产生的查询记录，并支持回到对应结果页。文章页面则提供科普文章列表浏览与详情阅读能力，用于增强系统的知识传播与持续学习功能。这样一来，系统不仅能够完成单次分类查询，也能在更长的时间尺度上形成“查询-理解-回看”的连续使用体验。

### 5.2 后端服务设计与实现

#### 5.2.1 Flask 工厂模式与蓝图组织

系统后端采用 Flask 工厂模式组织应用实例，在初始化阶段完成配置加载、蓝图注册、数据库初始化和扩展组件挂载，并预加载图像识别模型。用户、分类、知识图谱、文章和历史记录等业务按蓝图拆分组织，有助于控制模块耦合度并提高后续维护效率。对于本课题这类同时包含接口服务、模型推理和多业务模块的应用而言，工厂模式既有助于支持不同环境下的配置切换，也便于测试阶段快速构建独立应用实例。

#### 5.2.2 用户认证与 JWT 鉴权实现

用户登录后，后端根据微信 `code` 获取 `openid` 并生成 JWT 令牌。前端在后续请求中自动附带该令牌，后端通过鉴权装饰器完成解析与校验，并在校验未通过时返回统一错误信息。该机制在不明显增加前端调用复杂度的前提下，为历史记录等涉及用户数据的接口提供了基本保护。

#### 5.2.3 分类查询接口实现

文字搜索接口接收关键词并返回分页结果；图像识别接口负责文件校验、图片保存、识别服务调用和分类结果拼装；语音查询接口则负责接收音频数据、调用语音识别服务，并在识别成功后复用搜索服务返回结果。三类查询接口虽然在输入形式上存在差异，但在响应结构和错误处理策略上保持了较高一致性，这有助于前端统一处理不同类型的查询结果。

需要特别说明的是，语音服务在后端采用延迟加载 SDK 与单例客户端的实现方式。这样做的目的并非追求形式上的复杂设计，而是为了避免语音依赖影响其他模块的启动与测试，保证系统在未配置语音环境时仍能稳定运行其核心查询链路。

#### 5.2.4 知识图谱接口实现

知识图谱接口根据当前物品编号遍历知识关系数据，构建节点、边及分类数组，并输出与 ECharts 兼容的数据结构。这种接口设计使前端能够较低成本完成图谱渲染，同时保留节点联动和后续知识扩展的空间。换言之，知识图谱接口在系统中的作用不仅是“提供可视化数据”，更是支撑结果解释能力的关键接口。

#### 5.2.5 历史记录与文章接口实现

历史记录接口在鉴权通过后返回用户查询历史的分页结果，并统一承接图像、语音和文字三类查询记录；文章接口则提供列表与详情查询能力，并在文章详情访问时更新浏览次数。通过这两类接口，系统既能够记录用户行为轨迹，也能够增强内容阅读与知识传播的连续性，使应用从单次查询工具进一步延伸为持续学习平台。

### 5.3 图像识别模块实现

#### 5.3.1 数据集构建与预处理

系统使用 `train/val` 目录组织图像数据集，任务目标为可回收物、有害垃圾、厨余垃圾和其他垃圾四分类。训练阶段采用随机裁剪、翻转、旋转、颜色扰动和随机擦除等数据增强策略，以提升模型在复杂背景、不同光照条件和不同拍摄角度下的泛化能力。

#### 5.3.2 模型训练流程实现

系统采用基于 MobileNetV3 的迁移学习方案，加载预训练权重并替换分类头结构。项目训练脚本以 MobileNetV3-Large 为主实现训练流程，而在模型加载层保留了对不同变体权重的兼容能力。训练过程中进一步引入类别权重、标签平滑、AdamW 优化器、余弦退火调度和早停机制，以缓解类别不平衡问题并提高训练稳定性。

#### 5.3.3 模型加载与推理实现

后端通过单例方式加载模型和标签映射，避免重复初始化带来的额外开销。推理时先执行图像标准化处理，再输入模型计算类别概率，并保留前 3 个候选结果用于后续展示。这一设计一方面减少了服务层的重复加载成本，另一方面也为低置信度提示与候选结果说明提供了必要的数据支撑。

#### 5.3.4 低置信度提示机制实现

系统通过置信度阈值判断当前识别结果是否可靠。当置信度低于设定阈值时，结果页会提示“不确定”状态，并同步展示候选分类及相关物品，避免将单一低置信结果直接作为最终答案呈现给用户。相较于仅输出一个类别标签，这种做法更符合垃圾分类服务场景对结果稳健性和使用安全感的要求。

### 5.4 系统优化设计

#### 5.4.1 Redis 缓存机制实现

系统使用 Redis 作为热点结果缓存层，对搜索结果和图像识别结果进行临时缓存；当缓存服务不可用时，系统能够自动降级到数据库查询或模型推理流程，不影响主链路功能。该设计既有助于提升常见场景下的响应效率，也体现了系统在工程实现上对外部依赖故障的基本容错考虑。

#### 5.4.2 搜索性能优化实现

搜索服务采用“Redis 缓存 + MySQL 全文检索 + LIKE 兜底”的组合策略：优先命中缓存，未命中时先尝试全文检索，再通过模糊匹配补充结果，以兼顾查询性能与结果召回效果。当前实现中，搜索结果缓存的生存时间为 300 秒，这一设置较好适配了垃圾分类关键词在短时间内反复查询的使用特点。

#### 5.4.3 图像识别结果缓存实现

系统基于图片内容 MD5 构建缓存键，对重复上传的相同图片直接返回缓存结果，从而减少模型重复推理开销。当前实现中，图像识别结果缓存的生存时间为 3600 秒。该策略尤其适用于用户反复上传同一物品图片或进行联调测试的场景，能够有效降低重复推理带来的时间开销。

## 6 系统测试与结果分析

### 6.1 测试环境与测试方法

系统测试主要从模型效果、功能正确性、接口协同和系统性能四个方面展开，目标并非仅验证某一模块是否能够独立运行，而是尽可能评估系统在“算法输出、接口组织与业务闭环”三个层面上的综合实现质量。考虑到当前项目既包含可在本地完整验证的后端功能，也包含依赖微信环境和外部语音服务的链路，因此测试方法采用“自动化验证为主、阶段性手动联调为辅”的组合策略。

在自动化测试部分，项目主要采用 `pytest` 作为测试框架，并结合 Flask `test_client()`、`mock`/`patch` 等方式，对鉴权流程、图像识别服务、语音识别服务、搜索服务以及主要接口联动过程进行验证。对于依赖外部平台的部分，系统通过模拟响应方式验证业务逻辑、异常处理和回退路径，以提高测试结果的可重复性与稳定性。对于真实环境依赖较强的部分，则在论文中明确将其表述为“待后续联调补充验证”，以避免将尚未完成的工作误写为既有结论。

### 6.2 图像分类模型效果分析

根据项目训练与验收记录，系统图像分类模型在验证集上的准确率达到 85.02%。当前训练集规模为 46944 张，验证集规模为 5007 张。结合本课题的应用场景来看，该指标虽然仍存在进一步提升空间，但已经能够为垃圾四分类任务提供较稳定的识别支持，具备一定的工程应用基础。

### 6.3 功能测试

#### 6.3.1 用户登录测试

测试结果表明，用户登录后可以成功获取 JWT 并访问历史记录等受保护资源；未登录访问时，系统会返回明确的错误码和提示信息。这说明用户认证与鉴权链路运行正常，也说明系统在用户数据保护方面具备基础的访问控制能力。

#### 6.3.2 图像识别测试

图像识别测试已覆盖图片上传、格式校验、大小限制、历史记录写入和缓存命中等典型场景。根据项目记录，一次本地图像识别接口验收耗时约 0.0487 秒，说明在本地测试环境下，该链路具备较好的响应效率。对于面向即时投放场景的分类辅助系统而言，这种响应速度能够较好支撑“边查边分”的实际使用节奏。

#### 6.3.3 语音查询测试

语音识别服务测试覆盖了成功识别、接口错误、超时和历史写入等场景，说明后端语音服务链路已具备较好的可验证性。需要指出的是，当前前端侧仍以语音辅助输入为主，完整真机录音上传链路的稳定性仍有待后续联调进一步确认。因此，现阶段更准确的表述应是“语音辅助查询能力已实现，并完成了后端链路验证”，而不是认为语音功能在所有真实环境中都已完全验证。

#### 6.3.4 文字搜索测试

文字搜索已形成“关键词输入 -> 搜索接口 -> 结果列表 -> 详情查看”的完整闭环，并支持空结果返回与历史记录写入，能够满足用户在模糊查询、补充查询和兜底查询场景下的基本使用需求。就系统整体而言，文字搜索也是当前最稳定、覆盖面最广的基础查询方式。

#### 6.3.5 知识图谱与历史记录测试

知识图谱接口能够稳定返回节点和边结构，历史记录接口能够统一展示图像、语音和文字三种类型的查询记录，说明系统在“结果解释”和“行为留痕”两个方面已经形成较完整的数据链路。这也是系统区别于单一识别工具或单一搜索工具的重要特征之一。

### 6.4 接口测试与集成测试

根据项目开发记录，后端自动化测试当前共 42 项并已通过，集成测试覆盖了登录、搜索、详情、知识图谱、图片识别和历史记录等主要业务链路，同时也涉及未登录访问、空结果返回和无效物品编号等边界场景。这表明系统后端在接口组织、业务编排和异常处理方面已经具备较好的自动化验证基础。换言之，当前测试结论更适合被理解为“核心后端链路已得到验证”，而微信真机环境、真实语音联调和部署环境验证仍属于后续完善工作。

### 6.5 系统性能分析

系统通过 Redis 缓存优化搜索链路和图像识别链路。搜索服务在高频关键词场景下可直接命中缓存；图像识别服务在相同图片重复上传时能够跳过重复推理过程，从而减少响应时间和重复计算开销。结合当前实现可知，搜索缓存与图像缓存分别采用 300 秒和 3600 秒的生存时间设置，前者更适合短周期热点检索，后者更适合重复上传同一图像的场景。整体来看，缓存机制在不改变核心业务逻辑的前提下，提高了系统对重复查询场景的响应能力。

### 6.6 存在的问题与原因分析

尽管系统已经完成主要功能设计与实现，但从工程完善度和真实场景适配性来看，当前成果更应被界定为“阶段性完成核心功能验证”，而非所有应用条件下的最终定型版本。综合现有实现情况，仍存在以下不足：

1. 语音模块在前端侧仍以语音辅助输入为主，完整真机录音上传链路仍待进一步联调，因此语音能力在真实环境中的稳定性还需要继续验证。
2. 图像识别模型在复杂背景、遮挡场景和边缘样本场景下仍有进一步优化空间，当前识别效果尚不能覆盖所有复杂使用情形。
3. 知识库数据规模和地方化规则覆盖范围仍可继续扩展，现有知识内容更适合作为通用分类辅助，而非面向所有地区规则差异的完整方案。
4. 微信真机环境和 Docker 部署环境下的最终联调验证仍需进一步补充，当前结果更适合表述为“阶段性完成核心功能验证”。

## 7 总结与展望

### 7.1 全文总结

本文围绕居民在生活垃圾投放过程中“不会分、难查询、难理解”的现实问题，完成了一款智能垃圾分类助手小程序的设计与实现。论文从需求分析出发，构建了以前端微信小程序、后端 Flask 服务、MySQL 与 Redis 数据层、MobileNetV3 图像分类模型以及知识图谱展示模块为核心的整体方案，并实现了图像识别、语音辅助查询、文字搜索、知识图谱展示、科普文章浏览和历史记录管理等主要功能。

从论文工作内容看，本文较完整地完成了需求分析、总体设计、详细实现和测试分析四个层面的研究任务；从项目实现结果看，系统已经形成较清晰的业务闭环，并在后端自动化测试、主要接口联动和图像分类效果方面取得了阶段性结果。虽然部分真实环境验证工作仍需继续补充，但作为一项“设计与实现类”毕业设计课题，本文已经在问题分析、系统构建和阶段性验证方面达到了预期目标。

### 7.2 系统不足

系统当前在语音真机链路、复杂场景识别能力、知识库扩展以及部署联调等方面仍存在进一步完善空间。上述不足并不否定现有工作的阶段性成果，但也提示本文所完成的更多是一套具备实际运行基础的原型系统，其在复杂环境适配和工程完备性方面仍有继续深化的必要。

### 7.3 后续优化方向

结合现有实现情况，后续可从以下几个方面继续优化：

1. 完善语音录音上传与真机联调流程，提高语音能力在真实环境中的稳定性和可用性。
2. 扩充垃圾图像数据集，特别是复杂背景样本和长尾类别样本，进一步提升模型泛化能力。
3. 扩展知识库规模并补充地方化分类规则支持，使系统更贴近不同地区的实际投放规范。
4. 完善 Docker 部署、监控与日志管理机制，提高系统在部署和运维阶段的工程完整性。
5. 基于历史行为进一步探索个性化推荐和知识推送能力，提升系统持续使用价值。
